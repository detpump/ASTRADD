# Architect Mode Guidance
- Trading orchestration assumes a single event loop coordinating WebSocket ingestion, ML predictions, LLM reviews, and risk/self-heal operations ([`trading_system.py`](workspace/skills/aster-trading/src/trading_system.py:112)). Splitting these into separate services would break the contract with `centralized_logger` and RiskGuard’s shared state stored in `logs/risk_state.json`.
- There are three independent state stores (`trade_state.json`, `risk_state.json`, `v2_state.json`). Any architectural change must preserve this tri-state design or the dashboard, self-heal, and risk logic desynchronize. Use `sync_all_with_exchange()` and `RiskGuard.sync_with_exchange()` to keep them aligned.
- Risk parameters are recalibrated dynamically by the LLM adjuster ([`risk/dynamic_risk_adjuster.py`](workspace/skills/aster-trading/src/risk/dynamic_risk_adjuster.py:1)). Introducing static config layers or caching risk values elsewhere will be overwritten on the next adjustment cycle.
- External dependencies (API, LLM) are guarded by singleton circuit breakers in [`api/aster_api.py`](workspace/skills/aster-trading/src/api/aster_api.py:333). Replacing the HTTP layer must re-use those breakers or the trading loop loses its fail-safe behavior.
